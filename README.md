# 3-Agent Incident Response System

A modular, production-shaped pipeline that turns raw Linux API logs into an
operator-ready remediation playbook. Three agents collaborate over typed
Pydantic handoffs:

```
   ┌──────────────────────┐    ┌─────────────────────────┐    ┌─────────────────────┐
   │ Agent 1: Log Analyst │ →  │ Agent 2: Solution       │ →  │ Agent 3: Resolution │
   │ (gemini-2.5-flash)   │    │ Researcher (no LLM,     │    │ Planner             │
   │                      │    │ deterministic retrieval)│    │ (gemini-2.5-flash)  │
   └──────────────────────┘    └─────────────────────────┘    └─────────────────────┘
        Diagnosis                    ResearchBundle                  Playbook
```

## Setup

### 1. Create a virtualenv (recommended)

```bash
python3 -m venv .venv
source .venv/bin/activate          # macOS / Linux
# .\.venv\Scripts\activate         # Windows PowerShell
```

### 2. Install dependencies

```bash
pip install --upgrade pip
pip install pydantic google-generativeai duckduckgo-search
```

`pydantic` is the only hard dependency. `google-generativeai` enables the live
LLM path; `duckduckgo-search` enables Agent 2's optional live web augmentation.
Without either, the system still runs end-to-end via deterministic fallbacks.

### 3. Configure the API key

The key is read with:

```python
GEMINI_API_KEY = os.environ.get(
    "GEMINI_API_KEY",
    "AIzaSyAezDzgyKX5lOsjvrc3mhtATgaLVs3Kszw",
)
```

The fallback is only for evaluation convenience. **In any non-evaluation
environment, set the env var explicitly and remove the default** — see
`DEPLOYMENT_GUIDE.md` for secret-management options (AWS Secrets Manager,
GCP Secret Manager, Kubernetes Secrets, etc.):

```bash
export GEMINI_API_KEY="your-key-here"
```

## Run

```bash
# 1. Generate the synthetic logs (or supply your own under ./logs/)
python log_generator.py

# 2. Run the pipeline
python main.py --logs ./logs --out ./report.txt
```

CLI flags:

| Flag | Default | Purpose |
|------|---------|---------|
| `--logs DIR` | `./logs` | Directory containing the three log files |
| `--incident-id ID` | `INC-2026-04-26-001` | Stamp on the report |
| `--no-web` | off | Disable DuckDuckGo augmentation in Agent 2 |
| `--out PATH` | (none) | Also write the rendered report to a file |

## What you get

A four-section playbook (`pre_checks` → `remediation` → `post_checks` →
`rollback`), plus the diagnosis and reference list that produced it. See
`SAMPLE_OUTPUT.txt` for a captured run on the included logs — the system
correctly identifies SQLAlchemy `QueuePool` exhaustion driven by a session
leak introduced in deploy `2026.03.17-2`
(`portfolio/rebalance_service.py:118`), surfaces the cascade through Postgres
(`remaining connection slots are reserved`) and gunicorn (`Worker timeout`),
and recommends rolling back the offending deploy as the highest-leverage
mitigation.
